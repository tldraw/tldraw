01

Gather & steady the points

packages/freehand/src/lib/getStrokePoints.ts

Raw pointer input is hostile. Pens report pressure 0 while clearly touching the screen. The hand trembles. Points bunch up where you hesitate and pile on top of each other at the start and end of a gesture. Stage one’s job is to turn that mess into an evenly-behaved spine: a list of StrokePoints that each know their position, direction of travel, distance from the previous point, and running length.

The heart of it is streamlining — a one-pole low-pass filter. Each new point doesn’t go where the pointer is; it only travels a fraction t of the way there from the previous smoothed point, trailing the input like a pen dragged on a string. Higher streamline, longer string, smoother line, more lag.

z < 0.025 clamp, don’t delete — some pens and OSes report zero pressure while genuinely touching
dist²(p, first) ≤ (size/3)² strip points crowding the first point (keeping the max pressure seen) — same at the tail
i < 4 && totalLength < size noise gate: ignore the first few points until the stroke has actually travelled one stroke-width
point = lerp(input, prev, 1 − t),  t = 0.15 + (1 − streamline)·0.85 the streamline filter itself — at streamline 0, t = 1 and the input passes straight through
streamline, visualised raw input filtered spine
t = 0.15 + (1 − 0.50)·0.85 = 0.58
Each spine point moves only t of the way from the previous spine point toward the raw input. The wobble is high-frequency, so it averages away; the gesture is low-frequency, so it survives. When the stroke is complete, the last point is pinned to the true input so the line never falls short.

Two more quiet kindnesses live here: a two-point stroke drawn with simulated pressure gets four interpolated in-between points (otherwise fast taps make ugly “dash” marks), and a completed stroke appends a duplicate of its last point so the smoothing curve in stage five can catch all the way up to the end.

02

Pressure becomes radius

packages/freehand/src/lib/setStrokePointRadii.ts

Every stroke point now gets a radius — half the local width of the ink. The master formula maps pressure through the thinning option:

radius = size · easing(0.5 − thinning · (0.5 − pressure))

At thinning = 0 the radius is a constant size/2 and pressure is ignored. Positive thinning makes hard presses fat and light touches thin; negative thinning inverts that.

But where does pressure come from with a mouse? It’s simulated from speed. Big gaps between points mean a fast hand, and a fast hand — like a real pen — leaves a thinner line. The target pressure is 1 − min(1, distance/size), and the actual pressure chases that target at a deliberate lag (RATE_OF_PRESSURE_CHANGE = 0.275), so width changes are creamy, never jumpy. Real pen pressure is smoothed through the same chasing filter.

target = 1 − min(1, distance / size) simulated pressure: fast → thin, slow → thick
p += (target − p) · (sp · 0.275) pressure chases its target — the lag is what makes width changes feel like ink, not like a glitch
runningLength < size · 5 warm-up: pre-roll the pressure filter over the first stretch so strokes don’t start with a blob — “drawn lines almost always start slow!”
taper: radius ×= min(easeOutQuad(d/taperStart), easeOutCubic(rem/taperEnd)) optional tapers squeeze the ends toward a point (floored at 0.01 so the outline never collapses)
speed → pressure → width resulting ink pressure gap size (speed)

03

Split at the elbows

packages/freehand/src/lib/svgInk.ts — partitionAtElbows()

A single smooth outline can’t turn a hard corner — it rounds it off or ties itself in knots. So before drawing, svgInk hunts for elbows and cuts the stroke into partitions there. Each partition is rendered as its own closed subpath with its own round caps; two caps meeting at the elbow point is what makes corners look crisp and inky.

The detector is a dot product of the incoming and outgoing travel directions at each point: +1 means dead straight, 0 a right angle, −1 a full reversal. Two thresholds and one proximity test sort every point:

dpr < −0.8 always an elbow — and the cut happens at the raw input point, not the smoothed one, “for swooshiness in fast zaggy lines”
dpr > 0.7 comfortably smooth — keep going
else: (|in|² + |out|²) / r̄² < 1.5 a moderate angle only counts as an elbow if the point sits close to its neighbours relative to the local radius — a wide, slow turn stays smooth
elbow classifier — drag the points r̄ (local radius) shown as faint circles
 
 

After cutting, each partition is tidied: points crowding the new endpoints (within half the mean radius) are dropped, and the end points’ direction vectors are re-aimed at their nearest surviving neighbour — so every partition starts and ends with an honest heading for its caps.

04

Walk the outline

packages/freehand/src/lib/getStrokeOutlinePoints.ts — getStrokeOutlineTracks()

Now the spine becomes a shape. Walking point to point, the algorithm projects two offset points — one to the left, one to the right — perpendicular to the direction of travel, each exactly one radius away. The offset direction is blended between this point’s vector and the next one’s (weighted by their dot product), which pre-smooths the joints.

Not every candidate point is kept. Two filters thin the herd:

dist²(candidate, previous kept) ≤ (size · smoothing)² spacing filter: a side only accepts a new point once it has moved far enough along — higher smoothing, sparser tracks, rounder ink
within size · 0.05 of the chord, looking ≤ 8 points ahead simplifier: drop track points that the final curve wouldn’t miss — sub-0.1px difference, dramatically shorter paths
prevDpr < 0  ||  nextDpr < 0.2 a sharp turn inside a partition: if it’s gentler than −0.62 nudge both sides along the old heading (“soft corner”); otherwise sweep a half-circle of points around the raw input point (“sharp corner”)
left & right tracks spine left track right track · whiskers = radius offsets

05

Write the path

packages/freehand/src/lib/svgInk.ts — renderPartition() · fmt.ts

The tracks are still polylines — the curve appears in how they’re written. The path travels through the midpoints of consecutive track points using SVG’s smooth quadratic command t, which needs only one coordinate pair per segment (the control point is inferred by reflection). Down the left track, an arc (a) around the last point’s radius for the end cap, back up the right track in reverse, a second arc for the start cap, and Z. A single-point partition is just a circle.

Then the part nobody sees: this string is rebuilt on every frame while you draw, so it has to be cheap. Browsers can’t show differences below a hundredth of a pixel, so every coordinate is rounded to integer hundredths (“centi-pixels”) and the decimal digits are written directly into a byte buffer, skipping JavaScript’s slow float-to-string entirely. Commands are relative, so deltas stay small — and because deltas are differences of the rounded integers, they sum back exactly: no drift, ever.

midpoint quadratics — how a polyline becomes a curve track points (controls) midpoints (on the curve)
The curve only ever touches the midpoints; the track points act as control magnets. Chaining these guarantees the segments join with matching tangents — smooth by construction, no solver needed.
