# Context: marketing-assets

Glossary for the marketing-assets demo app. Terms only — no implementation detail.

## Marketing asset

The deliverable the user is producing (e.g. an Instagram square). A composite of a
[[Background]] image generated by an image model, at most one [[Text layer]] (a single
headline) the app renders deterministically on top, and the [[Accompanying copy]] shown
beside it. The image stays visually clean — one element at most on the pixels; everything
else the campaign says lives in the copy. The pixels come from the image model; the words
come from [[Plan]].

## Background

The text-free image behind an asset, generated and edited by the image model (default
Gemini 2.5 Flash Image). Carries the imagery, colour, and composition, with a calm area
left for the headline. Never contains rendered text — the single on-image headline is a
[[Text layer]]; the rest of the words are [[Accompanying copy]].

## Text layer

The single piece of copy drawn over the [[Background]] by the app — a headline (or one key
phrase): its words, position, size, font role, colour, alignment, and an optional contrast
scrim, all stored as data, not baked into pixels. Capped at one per asset so the image
never crowds; an asset may have none. Authored by [[Plan]], rendered deterministically, so
text stays legible, on-brand, and stable across re-renders. Distinct from the
[[Accompanying copy]], which sits beside the image rather than on it.

## Accompanying copy

The body text shown beside an asset — the social post / caption that runs with the image,
not rendered on it. Written by [[Plan]] in the brand tone and tailored to the [[Output type]]'s
platform and audience (e.g. business-focused for LinkedIn, developer-focused for X), at a
length that suits that platform. Stored per [[Version]], shown in a panel under the asset
with a copy button, and included in the [[Campaign export]]. This is where the message,
detail, and call to action live, keeping the image itself to a single [[Text layer]].

## Output type

A named target format chosen from a dropdown (Instagram square, story, landscape,
Facebook cover, X header, poster, …). Sets the [[Asset frame]] dimensions, a format hint
passed to the image model, and the platform that steers the voice and length of the
[[Accompanying copy]].

## Asset frame

A tldraw shape holding one [[Marketing asset]] — its [[Background]] and [[Text layer]]s —
and the [[Annotation]]s drawn on it. Created at the chosen [[Output type]]'s dimensions.
Multiple coexist on one canvas; each is annotated and versioned independently.

## Brand guidelines

A configurable set of constraints — colours, fonts (heading/body), tone, density, logo,
reference images — that shape how an asset is generated. Applied as prompt text plus
reference images, not as an exact deterministic render. User-editable through a sidebar
form (dropdowns, colour pickers). Singular and global: one brand applies to every
[[Asset frame]] in the session.

## Re-render

Producing a new [[Version]] of an asset from the current asset plus annotations.
[[Plan]] reads the annotations and updates the [[Text layer]]s directly (no image model),
handing back only the background changes; those are applied to the [[Background]] one at
a time. Distinct from a first generation, which starts from prompt + brand + references.

## Plan

The text stage, run by a vision LLM (default Claude, kept agent-agnostic — the provider and
model are chosen by the `PLAN_PROVIDER` / `PLAN_MODEL` env vars and resolved from a provider
registry). On a first generation it places a single headline [[Text layer]] over a fresh
[[Background]] and writes the [[Accompanying copy]]. On a [[Re-render]] it reads the
[[Annotated composite]], returns the updated headline and copy, and splits off any
background-only changes as instructions for the image model. Emits structured data, never
pixels.

## Annotated composite

A single image exported from an [[Asset frame]]: the current asset (background + text)
with its [[Annotation]] arrows and text marks visible. Input to [[Plan]] on a
[[Re-render]].

## Version

One generated state of an [[Asset frame]] — the [[Background]] image plus its
[[Text layer]]s, the instruction that produced it (or first-generation prompt), and a
timestamp. Renders append a version; reverting points "current" at an earlier version
non-destructively. Stored in the tldraw document. Separate from tldraw undo/redo, which
covers shape and [[Annotation]] edits.

## Annotation

A mark drawn on or around an asset to say what should change: something that points at a
spot and text that describes the change. Created either with standard tldraw arrows and
text, or with the dedicated [[Annotate tool]]. Read by [[Plan]] and never sent to the
image model as pixels. Re-render is triggered by an explicit button, not by drawing.

## Annotate tool

A toolbar tool that draws one [[Annotation]] as a single unit: a rectangle ringing the area to
change, an arrow pointing from a note to that rectangle, and the note's text, ready to type the
moment the rectangle is drawn. The three are ordinary tldraw shapes grouped together, so the
re-render pipeline reads them like any other [[Annotation]].

## Room

The shared space a canvas lives in, named by the room id in the URL. Everyone who opens the
link edits the same [[Marketing asset]]s, [[Brand guidelines]], [[Version]]s, and
[[Annotation]]s together in real time, with live cursors. The document is synced through a
server-hosted room and stored there, rather than in a single browser. Each visitor lands in
their own remembered room by default; sharing the link is what makes it collaborative.

## Asset upload

Where the bytes of a [[Background]] (and any media dropped on the canvas) live: uploaded to
the backend's object storage and referenced by URL on the synced document, so large images
don't travel through the [[Room]] inline. The image model still works on the raw bytes — the
background is read back from its URL whenever a [[Re-render]] needs to edit it.

## Brief

The campaign description the user writes, optionally sharpened with answered
[[Clarifying questions]]. Combined with the [[Brand guidelines]] and the chosen [[Output type]]
and sent to the models to produce a [[Batch]].

## Batch

A set of [[Marketing asset]]s generated together from one [[Brief]], laid out as a grid on the
canvas and filled in parallel. Each tile is nudged toward a different creative direction, and its
[[Accompanying copy]] toward a different messaging angle, so the batch explores a range and every
asset's caption is unique rather than repeating one idea.

## Verdict

A reviewer's like or dislike on an idea, stored on its [[Asset frame]] and shown as a coloured
ring. Liked ideas are fed back as references to the next batch; disliked ones steer it away.

## Refine round

A follow-up [[Batch]] generated from the current [[Verdict]]s plus written feedback: approved
ideas become references to riff on, rejected directions are avoided. Placed below the previous
rounds so the canvas reads as a top-to-bottom history of the campaign.

## Selection variation

A [[Batch]] seeded by the current canvas selection — the selected shapes are exported to one
image and passed as the key visual reference, so a region the user likes can spin off new ideas.

## Clarifying questions

A short list of questions the [[Plan]] stage can suggest from the [[Brief]], brand, and format
before the first [[Batch]]. Answering them folds extra detail into the brief; skipping them is
fine.

## Company brand

The reusable [[Brand guidelines]] saved outside any one [[Room]] (in the browser's local
storage), used to seed a fresh room so a new campaign starts from the last brand configured
rather than the built-in defaults.

## Campaign export

A downloadable bundle of chosen assets — liked, selected, or all — with a full-resolution image
per [[Marketing asset]], each asset's [[Accompanying copy]] as a text file, and a spreadsheet of
every [[Text layer]] and caption, for handing to an ad platform.
