Why exec timed out for 30 seconds — and how the rendezvous fixes it

Every exec call runs JavaScript on a tldraw canvas inside the host's widget iframe. The server used to block on an in-memory promise until the widget reported back — a design that silently assumed the host would route that report to the same server instance. Claude doesn't.

The symptom

In Claude, the canvas rendered and the code visibly ran, but the tool call still hung for the full timeout and handed the model an error:

Exec failed: Callback timed out after 30000ms for channel "exec"

The model would then apologize, retry, and time out again — even though the drawing was sitting right there, finished.

Why it happened

Instrumented logs (mcp-session-id on every request) showed each host has a different idea of "the session". The pending promise was keyed by session, so correctness depended entirely on host behavior we don't control:

Host Model tool calls Widget's callback Result
Cursor One persistent session Same session → same DO Resolved inline. Worked.
Claude Fresh session per call A separate long-lived session → always a different DO Callback dropped as "no pending request". Guaranteed timeout.

So the callback wasn't slow or lost — it arrived within ~1.6 seconds, every time, at a server instance that had never heard of the exec call waiting elsewhere.

The fix

  • Rendezvous keyed by the code itself The one value both sides share — regardless of session routing — is the exec code string. Server and widget independently derive execKey = sha256(code), which names a dedicated rendezvous DO. A callback landing on the wrong session DO forwards the result there; the blocked exec races its local promise against the rendezvous and takes whichever answers first. Early arrivals are stashed, so ordering doesn't matter either.
  • A bounded, graceful wait The 30s hang became a 4s window (8s for code editors, whose cold-start callbacks measured ~4.3s). If nothing arrives, exec returns a non-error "your code is executing" result — the widget pushes canvas state to the model's context on its own, so a missed handoff degrades instead of failing.
  • The canvasId travels in the tool result New canvases only learned their server-assigned id from the callback response — the exact channel that misroutes. It now rides structuredContent on every exec result, which the host reliably delivers to the widget, keeping multi-turn "edit the same canvas" working.
  • Two widget bugs found along the way Claude's event ordering made every exec run its code twice (a debounced run surviving the run-once guard reset), and models passing arrays to variadic methods like select([...]) crashed the proxy. Both fixed.
The moving parts live in apps/mcp-app/src: the rendezvous DO in canvas-store.ts, the shared key derivation in shared/exec-key.ts, the bounded wait and race in tools/exec.ts, and the callback forwarding in register-tools.ts.
