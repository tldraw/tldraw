A plugin bundles custom records, shapes, tools, and UI into a single package that you pass to `<Tldraw>` and, if you're using sync, to `useSync` and your server. tldraw ships one plugin today, `@tldraw/comments`, which adds comment threads to the canvas.

## What a plugin is

On the client, a plugin is a `TldrawPlugin` object:

```tsx
interface TldrawPlugin {
	id: string
	records?: Record<string, CustomRecordInfo>
	shapeUtils?: readonly TLAnyShapeUtilConstructor[]
	tools?: readonly TLStateNodeConstructor[]
	components?: TLComponents
	overrides?: TLUiOverrides
	onMount?(editor: Editor): void | (() => void)
}
```

Every field is optional except `id`, which must be unique among the plugins you register - duplicate ids throw when the editor sets up. Pass one or more plugins to `<Tldraw>`:

```tsx
import { commentsPlugin } from '@tldraw/comments'
import { Tldraw } from 'tldraw'

const plugins = [commentsPlugin()]

export default function App() {
	return <Tldraw plugins={plugins} />
}
```

Pass a stable array (module-level, `useMemo`, or `useState`) rather than an inline literal, since a new array on every render would force the editor to rebuild its shape utils, tools, and schema.

### What plugins can contribute

- **`records`** - custom record types, keyed by type name. These get merged into the store's schema alongside the default tldraw records.
- **`shapeUtils`** and **`tools`** - registered with the editor the same way the ones you pass directly to `<Tldraw>` are.
- **`overrides`** - [UI overrides](/docs/user-interface), merged with your own overrides and other plugins' in registration order.
- **`onMount`** - called when the editor mounts. Plugin `onMount` handlers run first, in plugin order, then yours. If a handler returns a cleanup function, cleanups run in the same order when the editor unmounts.
- **`components`** - see below.

### Component slots: stacking vs. overriding

Most [component slots](/docs/user-interface) are non-stackable: the first plugin that sets a slot wins among the plugins, but your own `components` prop always wins overall - including passing `null` to hide a slot a plugin set.

Two slots are stackable: `InFrontOfTheCanvas` and `OnTheCanvas`. Every plugin's contribution to these slots renders, in plugin order, followed by yours if you provide one. This is how the comments plugin can render its pins and popovers without stepping on a custom `InFrontOfTheCanvas` component you supply - both render together. Passing `null` for a stackable slot in your own `components` hides it entirely, including every plugin's contribution.

Plugin overlays in `InFrontOfTheCanvas` and `OnTheCanvas` render even when the editor is mounted with `hideUi`, since they're canvas overlays rather than chrome.

## The comments package

`@tldraw/comments` adds comment threads to the canvas: pins anchored to shapes, points, regions, or pages, with a popover for reading and a composer for writing.

### Client setup

```tsx
import { commentsPlugin } from '@tldraw/comments'
import { Tldraw } from 'tldraw'

const plugins = [commentsPlugin()]

export default function App() {
	return <Tldraw plugins={plugins} />
}
```

This registers the `comment` and `comment-thread` record types and renders the default UI (pins, popover, composer) through the `InFrontOfTheCanvas` slot. `commentsPlugin` takes two optional options:

```tsx
interface CommentsPluginOptions {
	user?(editor: Editor): CommentsPluginUser | null
	components?: Partial<TLCommentsComponents>
}
```

`user` resolves the identity used to author new comments. By default it falls back to the editor's user preferences identity, which always exists - including a locally generated, anonymous id for a solo session. **This default is not auth-gated.** If you want to restrict who can compose comments in your app, pass your own `user` function and return `null` to disable the composer for a given editor state. If you're using tldraw sync, the server additionally gates comment writes per session (see below), so client-side gating is about UX, not security.

`components` lets you override the pin or composer components while keeping the rest of the default UI. The thread popover is rendered by the pin component, so overriding the pin also replaces the popover.

### Reading and writing comments

For custom UI, `@tldraw/comments` exports headless hooks and functions instead of requiring you to use the default components:

| Export                    | Description                                                      |
| ------------------------- | ---------------------------------------------------------------- |
| [`useCommentThreads`](?)  | Reactive list of comment threads, optionally filtered to a page. |
| [`useThreadComments`](?)  | Reactive, `createdAt`-ordered comments for a thread.             |
| [`startCommentThread`](?) | Creates a thread with its first comment.                         |
| [`addComment`](?)         | Appends a comment to an existing thread.                         |
| [`resolveThread`](?)      | Marks a thread resolved.                                         |

All of these mutate the store directly and are **not undoable** - they run with `{ history: 'ignore' }` so a comment doesn't reappear in someone else's session after an undo, and doesn't get wiped out by someone else's redo in a multiplayer document.

### Server setup with sync

If you're using tldraw sync, register the matching server-side plugin so comment records are part of the schema and sync between clients:

```tsx
import { commentsSyncPlugin } from '@tldraw/comments/server'
import { TLSocketRoom } from '@tldraw/sync-core'

const room = new TLSocketRoom({
	plugins: [commentsSyncPlugin()],
})
```

`commentsSyncPlugin` accepts an `onChange` callback if you want to observe comment writes, for example to project them into an app database:

```tsx
import { commentsSyncPlugin } from '@tldraw/comments/server'

const plugin = commentsSyncPlugin({
	onChange({ puts, deletes }) {
		// puts: comments and threads that were created or updated
		// deletes: ids of comments and threads that were removed
	},
})
```

`onChange` only fires for commits that actually touch comment records, and only receives the comment-related puts and deletes from the commit - not the rest of the document diff.

Comments are served through sync's object-store lane rather than the main document lane: they're gated per session by `objectAccess` (`'read'` or `'write'`) instead of `isReadonly`, so a session can be able to comment without being able to edit the document, and vice versa. Object-store records are also excluded from document snapshots and `.tldr` exports on the server.

## Consumption tiers

You can adopt comments at three levels of integration, and move between them without changing the record shape:

### 1. Local only

Register `commentsPlugin()` on `<Tldraw>` with no sync setup at all. Comment records persist to IndexedDB with the rest of the document via `persistenceKey`, and survive reloads. This is the fastest way to try comments out; see the [comments plugin example](https://github.com/tldraw/tldraw/tree/main/apps/examples/src/examples/configuration/comments-plugin) for a working setup.

### 2. With sync

Register `commentsPlugin()` on the client and `commentsSyncPlugin()` on the server. Comments now sync between connected clients like any other collaborative data, and the per-session `objectAccess` setting controls who can write them - for example, anonymous viewers get `'read'` while authenticated users get `'write'`.

### 3. External projection

Add an `onChange` callback to `commentsSyncPlugin` to push comment writes into your own database as they happen. This is useful for building views outside the canvas - a cross-file comment inbox, notifications, or moderation tooling - that need to read comments without opening the document itself.

## Plugins and schema compatibility

Plugin records are still document records, so ordinary [migration](/docs/persistence#migrations) and schema rules apply, with a few consequences specific to plugins:

- **Guard migrations, not down-migrations.** A plugin's record types ship with a retroactive identity migration and no `down` step. If an older client without the plugin connects to a server whose schema includes it, the server can't safely down-migrate plugin records into a schema that doesn't know the type at all, so the connection fails closed with `CLIENT_TOO_OLD` (prompting the client to refresh) instead of silently corrupting data.
- **Plugins must match on both ends.** A client that registers a plugin against a server that doesn't (or vice versa) fails to connect. Adding or removing a plugin is a schema change - register it on both the client (`useSync({ plugins })` or `<Tldraw plugins={[...]}>`) and the server (`TLSocketRoom({ plugins })`, and the storage constructor if you build the schema from `plugins` there too).
- **Evolve records by extending the migration sequence.** When a plugin's record shape needs to change, add a new step to that plugin's migration sequence rather than editing the identity guard migration or changing the record type's validator in place.
- **Deploy the server first.** As with any sync schema change, roll out the updated server before the updated client so there's never a window where clients speak a schema the server doesn't recognize.

## Writing your own plugin

A plugin for your own record types follows the same shape as `commentsPlugin`. On the client, bundle your `records`, any `shapeUtils` or `tools` that operate on them, and the components that render them:

```tsx
import { TldrawPlugin } from 'tldraw'

function myPlugin(): TldrawPlugin {
	return {
		id: 'my-app.annotations',
		records: myAnnotationSchemaRecords,
		components: {
			InFrontOfTheCanvas: MyAnnotationsOverlay,
		},
	}
}
```

If you're using sync, mirror the record registration in a `TLSyncPlugin` for the server:

```tsx
import { TLSyncPlugin } from '@tldraw/sync-core'

function myServerPlugin(): TLSyncPlugin {
	return {
		id: 'my-app.annotations',
		records: myAnnotationSchemaRecords,
		objectTypes: ['my-annotation'], // omit if your records belong on the main document lane
	}
}
```

Only set `objectTypes` for record types you want served through the object-store lane, gated by `objectAccess` rather than `isReadonly` and excluded from document snapshots - most custom records should stay on the main document lane and can omit this field entirely.
